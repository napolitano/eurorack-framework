# 74HC595 driver

## Purpose

`eurorack-driver-74hc595` is the software driver for **74hc595**. This page documents the framework-facing contract. Device electrical behavior and board-level integration are documented separately in [docs/hardware/shift-register/74hc595.md](../hardware/shift-register/74hc595.md).

## Public API

- Public header: `eurorack/drivers/shift/shift_register_74hc595.hpp`
- Primary type: `ShiftRegister74Hc595`
- Maturity: `alpha`

- `setOutput()` changes one buffered bit and validates its index.
- `clear()` clears the local buffer without bus traffic.
- `flush()` writes the complete chain and pulses the latch.
- `setEnabled()` controls optional active-low OE.

The generated Doxygen reference remains authoritative for exact signatures, parameter types, enumerators, and return annotations.

## Lifecycle and configuration

Construct with SPI, the storage-register latch, a device count, and optionally active-low output enable. Modify the local buffer and commit it with `flush()`.

Constructors do not transfer ownership of framework adapters. Configuration values are captured by value unless the public header explicitly accepts a pointer or reference. No implicit global initialization is performed.

## Ownership and memory

The driver owns a `std::vector` output buffer sized during construction. SPI and GPIO dependencies are borrowed and must outlive it.

Copy and move behavior follows the declarations in the public header. Applications should normally create one long-lived driver instance per physical device or bus address.

## Error handling

Mutating an out-of-range output returns `IoResult::InvalidArgument`. `flush()` returns the SPI result; the local buffer remains available for retry.

Callers shall inspect every returned `eurorack::io::IoResult` or `AnalogSample::result`. The driver does not log, throw exceptions, or silently perform unbounded retries.

## ISR, reentrancy, and concurrency

This driver is not reentrant. Foreground use only. No internal locking or atomic update protection is provided. Do not mutate or flush the same instance concurrently.

Unless a future API explicitly states otherwise, this library makes no thread-safety guarantee. Shared buses must be arbitrated above the driver layer.

## Dependencies and resources

The library depends only on the granular framework interfaces declared by its `library.json`. Hardware resources are represented by injected adapters rather than selected globally. Refer to the [hardware page](../hardware/shift-register/74hc595.md) and the framework resource registry for concrete bus and GPIO requirements.

## Usage example

The executable example is located at `examples/components/shift-register-74hc595`:

```cpp
#include "../driver_example_support.hpp"
#include <eurorack/drivers/shift/shift_register_74hc595.hpp>
int main(){ example::SpiBus spi; example::DigitalOutput latch,oe; eurorack::drivers::shift::ShiftRegister74Hc595 outputs(spi,latch,2U,&oe); if(outputs.setOutput(9U,true)!=eurorack::io::IoResult::Success) return 1; outputs.setEnabled(true); return outputs.flush()==eurorack::io::IoResult::Success && outputs.output(9U) ? 0:1; }
```

The repository example, not this excerpt, is the build-checked source of truth.

## Limitations

- No internal scheduler, locking, or background transfer engine is provided.
- Electrical limits, decoupling, level compatibility, and analog accuracy remain hardware-design responsibilities.
- Unsupported silicon features are listed on the hardware page.
- API compatibility is not yet guaranteed because the framework remains alpha.
