# DAC8568 driver

## Purpose

`eurorack-driver-dac8568` is the software driver for **dac8568**. This page documents the framework-facing contract. Device electrical behavior and board-level integration are documented separately in [docs/hardware/dac/dac8568.md](../hardware/dac/dac8568.md).

## Public API

- Public header: `eurorack/drivers/dac/dac8568.hpp`
- Primary type: `Dac8568`
- Maturity: `alpha`

- `setCode()` and `code()` manage eight buffered 16-bit channel codes.
- `writeInput()`, `writeAndUpdate()`, and `writeAndUpdateAll()` expose device update modes.
- `flushAll()` commits all buffered channels.
- Power, internal reference, reset, clear-code, and LDAC configuration commands are supported.

The generated Doxygen reference remains authoritative for exact signatures, parameter types, enumerators, and return annotations.

## Lifecycle and configuration

Construct with SPI and active-low SYNC. Store channel codes locally, then choose an input-register, immediate-update, or synchronized multi-channel transfer operation.

Constructors do not transfer ownership of framework adapters. Configuration values are captured by value unless the public header explicitly accepts a pointer or reference. No implicit global initialization is performed.

## Ownership and memory

SPI and chip-select are non-owning references. The driver owns only its fixed eight-channel code array and does not allocate dynamically.

Copy and move behavior follows the declarations in the public header. Applications should normally create one long-lived driver instance per physical device or bus address.

## Error handling

Command methods return `IoResult` and propagate SPI failures. Invalid channel use is validated where the public operation requires a physical channel.

Callers shall inspect every returned `eurorack::io::IoResult` or `AnalogSample::result`. The driver does not log, throw exceptions, or silently perform unbounded retries.

## ISR, reentrancy, and concurrency

This driver is not reentrant. Not internally synchronized or reentrant. Keep bus arbitration outside the driver and avoid concurrent calls from ISR and foreground contexts.

Unless a future API explicitly states otherwise, this library makes no thread-safety guarantee. Shared buses must be arbitrated above the driver layer.

## Dependencies and resources

The library depends only on the granular framework interfaces declared by its `library.json`. Hardware resources are represented by injected adapters rather than selected globally. Refer to the [hardware page](../hardware/dac/dac8568.md) and the framework resource registry for concrete bus and GPIO requirements.

## Usage example

The executable example is located at `examples/components/dac8568`:

```cpp
#include "../driver_example_support.hpp"
#include <eurorack/drivers/dac/dac8568.hpp>
int main(){ example::SpiBus spi; example::DigitalOutput sync; eurorack::drivers::dac::Dac8568 dac(spi,sync); dac.setCode(eurorack::drivers::dac::Dac8568Channel::A,32768U); const auto r=dac.writeAndUpdate(eurorack::drivers::dac::Dac8568Channel::A); return r==eurorack::io::IoResult::Success && spi.transfers()==1U ? 0:1; }
```

The repository example, not this excerpt, is the build-checked source of truth.

## Limitations

- No internal scheduler, locking, or background transfer engine is provided.
- Electrical limits, decoupling, level compatibility, and analog accuracy remain hardware-design responsibilities.
- Unsupported silicon features are listed on the hardware page.
- API compatibility is not yet guaranteed because the framework remains alpha.
