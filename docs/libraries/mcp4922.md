# MCP4922 driver

## Purpose

`eurorack-driver-mcp4922` is the software driver for **mcp4922**. This page documents the framework-facing contract. Device electrical behavior and board-level integration are documented separately in [docs/hardware/dac/mcp4922.md](../hardware/dac/mcp4922.md).

## Public API

- Public header: `eurorack/drivers/dac/mcp4922.hpp`
- Primary type: `Mcp4922`
- Maturity: `alpha`

- Two 12-bit channels with external references.
- Per-channel gain, reference-buffer, and shutdown bits.
- Configurable SPI clock and command framing.
- Optional LDAC-assisted simultaneous update.

The generated Doxygen reference remains authoritative for exact signatures, parameter types, enumerators, and return annotations.

## Lifecycle and configuration

Construct with SPI, CS, optional LDAC, and `Mcp4922Config`. Set buffered code, gain, reference-buffer, and enable state before flushing channels.

Constructors do not transfer ownership of framework adapters. Configuration values are captured by value unless the public header explicitly accepts a pointer or reference. No implicit global initialization is performed.

## Ownership and memory

SPI and GPIO dependencies are borrowed and must outlive the object. Per-channel state uses fixed arrays; no dynamic allocation occurs.

Copy and move behavior follows the declarations in the public header. Applications should normally create one long-lived driver instance per physical device or bus address.

## Error handling

Transfer functions return `IoResult` and preserve buffered state after failures so the caller may retry. Out-of-range code bits are constrained by frame generation.

Callers shall inspect every returned `eurorack::io::IoResult` or `AnalogSample::result`. The driver does not log, throw exceptions, or silently perform unbounded retries.

## ISR, reentrancy, and concurrency

This driver is not reentrant. Synchronous, non-reentrant, and not thread-safe. Shared bus ownership and ISR exclusion are caller responsibilities.

Unless a future API explicitly states otherwise, this library makes no thread-safety guarantee. Shared buses must be arbitrated above the driver layer.

## Dependencies and resources

The library depends only on the granular framework interfaces declared by its `library.json`. Hardware resources are represented by injected adapters rather than selected globally. Refer to the [hardware page](../hardware/dac/mcp4922.md) and the framework resource registry for concrete bus and GPIO requirements.

## Usage example

The executable example is located at `examples/components/mcp4922`:

```cpp
#include "../driver_example_support.hpp"
#include <eurorack/drivers/dac/mcp4922.hpp>
int main(){ example::SpiBus spi; example::DigitalOutput cs,ldac; eurorack::drivers::dac::Mcp4922 dac(spi,cs,&ldac); dac.setCode(eurorack::drivers::dac::Mcp4922Channel::A,2048U); dac.setGain(eurorack::drivers::dac::Mcp4922Channel::A,eurorack::drivers::dac::Mcp4922Gain::OneX); const auto r=dac.flushBoth(); return r==eurorack::io::IoResult::Success && spi.transfers()==2U ? 0:1; }
```

The repository example, not this excerpt, is the build-checked source of truth.

## Limitations

- No internal scheduler, locking, or background transfer engine is provided.
- Electrical limits, decoupling, level compatibility, and analog accuracy remain hardware-design responsibilities.
- Unsupported silicon features are listed on the hardware page.
- API compatibility is not yet guaranteed because the framework remains alpha.
