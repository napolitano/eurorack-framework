# TLC5916 driver

## Purpose

`eurorack-driver-tlc5916` is the software driver for **tlc5916**. This page documents the framework-facing contract. Device electrical behavior and board-level integration are documented separately in [docs/hardware/led/tlc5916.md](../hardware/led/tlc5916.md).

## Public API

- Public header: `eurorack/drivers/led/tlc5916.hpp`
- Primary type: `Tlc5916`
- Maturity: `alpha`

- Eight constant-current sink outputs per device.
- Implements the framework `LedBank` contract.
- Buffered output changes and explicit transfer.
- Supports chain-oriented serial transfer according to the driver constructor.

The generated Doxygen reference remains authoritative for exact signatures, parameter types, enumerators, and return annotations.

## Lifecycle and configuration

Construct with SPI and the required latch/output-control GPIO dependencies. Set logical LED states through the `LedBank` interface and flush the output register.

Constructors do not transfer ownership of framework adapters. Configuration values are captured by value unless the public header explicitly accepts a pointer or reference. No implicit global initialization is performed.

## Ownership and memory

Bus and GPIO dependencies are borrowed. Internal state is fixed-size for the configured abstraction and no ownership of LED hardware is implied.

Copy and move behavior follows the declarations in the public header. Applications should normally create one long-lived driver instance per physical device or bus address.

## Error handling

Mutators validate channel indices through the `LedBank` contract. Transfer methods propagate `IoResult` from SPI.

Callers shall inspect every returned `eurorack::io::IoResult` or `AnalogSample::result`. The driver does not log, throw exceptions, or silently perform unbounded retries.

## ISR, reentrancy, and concurrency

This driver is not reentrant. Foreground-only, non-reentrant API. Output updates and shared SPI access require external serialization.

Unless a future API explicitly states otherwise, this library makes no thread-safety guarantee. Shared buses must be arbitrated above the driver layer.

## Dependencies and resources

The library depends only on the granular framework interfaces declared by its `library.json`. Hardware resources are represented by injected adapters rather than selected globally. Refer to the [hardware page](../hardware/led/tlc5916.md) and the framework resource registry for concrete bus and GPIO requirements.

## Usage example

The executable example is located at `examples/components/tlc5916`:

```cpp
#include "../driver_example_support.hpp"
#include <eurorack/drivers/led/tlc5916.hpp>
int main(){ example::SpiBus spi; example::DigitalOutput latch,oe; eurorack::drivers::led::Tlc5916 leds(spi,latch,1U,&oe); if(leds.setBrightness(2U,65535U)!=eurorack::io::IoResult::Success) return 1; return leds.flush()==eurorack::io::IoResult::Success && leds.brightness(2U)!=0U ? 0:1; }
```

The repository example, not this excerpt, is the build-checked source of truth.

## Limitations

- No internal scheduler, locking, or background transfer engine is provided.
- Electrical limits, decoupling, level compatibility, and analog accuracy remain hardware-design responsibilities.
- Unsupported silicon features are listed on the hardware page.
- API compatibility is not yet guaranteed because the framework remains alpha.
