# MCP23017 driver

## Purpose

`eurorack-driver-mcp23017` is the software driver for **mcp23017**. This page documents the framework-facing contract. Device electrical behavior and board-level integration are documented separately in [docs/hardware/gpio/mcp23017.md](../hardware/gpio/mcp23017.md).

## Public API

- Public header: `eurorack/drivers/gpio/mcp23017.hpp`
- Primary type: `Mcp23017`
- Maturity: `alpha`

- Provides port direction, pull-up, GPIO read, and output-latch operations.
- Uses the framework I²C abstraction rather than Arduino Wire directly.
- Keeps protocol state minimal; register values are supplied explicitly by the caller.

The generated Doxygen reference remains authoritative for exact signatures, parameter types, enumerators, and return annotations.

## Lifecycle and configuration

Construct with an I²C bus and device address. Configure direction and pull-ups, then read or write GPIO ports through the register-level API.

Constructors do not transfer ownership of framework adapters. Configuration values are captured by value unless the public header explicitly accepts a pointer or reference. No implicit global initialization is performed.

## Ownership and memory

The I²C bus is borrowed and must outlive the driver. The object does not own external pins or interrupts and does not allocate dynamically.

Copy and move behavior follows the declarations in the public header. Applications should normally create one long-lived driver instance per physical device or bus address.

## Error handling

All bus operations return `IoResult`. Invalid addresses or backend failures are reported by the called operation; the driver does not retry automatically.

Callers shall inspect every returned `eurorack::io::IoResult` or `AnalogSample::result`. The driver does not log, throw exceptions, or silently perform unbounded retries.

## ISR, reentrancy, and concurrency

This driver is not reentrant. Foreground use only unless the supplied I²C backend explicitly supports ISR access and external serialization. The driver itself is not reentrant.

Unless a future API explicitly states otherwise, this library makes no thread-safety guarantee. Shared buses must be arbitrated above the driver layer.

## Dependencies and resources

The library depends only on the granular framework interfaces declared by its `library.json`. Hardware resources are represented by injected adapters rather than selected globally. Refer to the [hardware page](../hardware/gpio/mcp23017.md) and the framework resource registry for concrete bus and GPIO requirements.

## Usage example

The executable example is located at `examples/components/mcp23017`:

```cpp
#include "../driver_example_support.hpp"
#include <eurorack/drivers/gpio/mcp23017.hpp>
int main(){ example::I2cBus bus; eurorack::drivers::gpio::Mcp23017 gpio(bus,eurorack::io::I2cAddress{0x20U}); if(gpio.initialize()!=eurorack::io::IoResult::Success) return 1; if(gpio.setDirection(0xFF00U)!=eurorack::io::IoResult::Success) return 1; return gpio.writePin(0U,true)==eurorack::io::IoResult::Success ? 0:1; }
```

The repository example, not this excerpt, is the build-checked source of truth.

## Limitations

- No internal scheduler, locking, or background transfer engine is provided.
- Electrical limits, decoupling, level compatibility, and analog accuracy remain hardware-design responsibilities.
- Unsupported silicon features are listed on the hardware page.
- API compatibility is not yet guaranteed because the framework remains alpha.
