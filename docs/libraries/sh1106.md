# SH1106 display driver

## Purpose

`eurorack-driver-sh1106` is the software driver for **sh1106**. This page documents the framework-facing contract. Device electrical behavior and board-level integration are documented separately in [docs/hardware/display/sh1106.md](../hardware/display/sh1106.md).

## Public API

- Public header: `eurorack/drivers/display/sh1106.hpp`
- Primary type: `Sh1106`
- Maturity: `alpha`

- Supports monochrome page-oriented SH1106 displays.
- Uses caller-provided framebuffer storage.
- Separates controller initialization from framebuffer transfer.
- Configuration covers address and geometry-related settings exposed by the header.

The generated Doxygen reference remains authoritative for exact signatures, parameter types, enumerators, and return annotations.

## Lifecycle and configuration

Construct with I²C, configuration, and a caller-owned framebuffer. Initialize the controller, update the framebuffer through drawing code, then transfer it with the display update method.

Constructors do not transfer ownership of framework adapters. Configuration values are captured by value unless the public header explicitly accepts a pointer or reference. No implicit global initialization is performed.

## Ownership and memory

I²C and framebuffer memory are borrowed. Their lifetime must exceed the display object lifetime. The driver does not allocate the framebuffer.

Copy and move behavior follows the declarations in the public header. Applications should normally create one long-lived driver instance per physical device or bus address.

## Error handling

Initialization and transfer methods return `IoResult`. A failed transfer does not alter caller-owned framebuffer contents and may be retried.

Callers shall inspect every returned `eurorack::io::IoResult` or `AnalogSample::result`. The driver does not log, throw exceptions, or silently perform unbounded retries.

## ISR, reentrancy, and concurrency

This driver is not reentrant. Foreground use only. Initialization and full-buffer updates are blocking bus operations and are not reentrant.

Unless a future API explicitly states otherwise, this library makes no thread-safety guarantee. Shared buses must be arbitrated above the driver layer.

## Dependencies and resources

The library depends only on the granular framework interfaces declared by its `library.json`. Hardware resources are represented by injected adapters rather than selected globally. Refer to the [hardware page](../hardware/display/sh1106.md) and the framework resource registry for concrete bus and GPIO requirements.

## Usage example

The executable example is located at `examples/components/sh1106`:

```cpp
#include "../driver_example_support.hpp"
#include <array>
#include <eurorack/drivers/display/sh1106.hpp>
int main(){ example::I2cBus bus; eurorack::drivers::display::Sh1106 display(bus); std::array<std::uint8_t,1024U> pixels{}; eurorack::display::MonochromeCanvas canvas(pixels.data(),pixels.size(),128,64); canvas.applyPixel({10,10},eurorack::display::PixelOperation::Set); if(display.initialize()!=eurorack::io::IoResult::Success) return 1; return display.flush(canvas)==eurorack::io::IoResult::Success ? 0:1; }
```

The repository example, not this excerpt, is the build-checked source of truth.

## Limitations

- No internal scheduler, locking, or background transfer engine is provided.
- Electrical limits, decoupling, level compatibility, and analog accuracy remain hardware-design responsibilities.
- Unsupported silicon features are listed on the hardware page.
- API compatibility is not yet guaranteed because the framework remains alpha.
