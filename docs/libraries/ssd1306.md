# SSD1306 display driver

## Purpose

`eurorack-driver-ssd1306` is the software driver for **ssd1306**. This page documents the framework-facing contract. Device electrical behavior and board-level integration are documented separately in [docs/hardware/display/ssd1306.md](../hardware/display/ssd1306.md).

## Public API

- Public header: `eurorack/drivers/display/ssd1306.hpp`
- Primary type: `Ssd1306`
- Maturity: `alpha`

- Supports configured 128-pixel-wide 32- or 64-pixel-high monochrome displays.
- Caller-owned framebuffer avoids hidden allocation.
- Controller initialization and full framebuffer transfer are explicit.

The generated Doxygen reference remains authoritative for exact signatures, parameter types, enumerators, and return annotations.

## Lifecycle and configuration

Construct with I²C, `Ssd1306Config`, and caller-owned framebuffer storage. Initialize once, draw into the buffer, and issue an update to send complete pages.

Constructors do not transfer ownership of framework adapters. Configuration values are captured by value unless the public header explicitly accepts a pointer or reference. No implicit global initialization is performed.

## Ownership and memory

The I²C bus and framebuffer are borrowed. The caller controls storage lifetime and drawing synchronization.

Copy and move behavior follows the declarations in the public header. Applications should normally create one long-lived driver instance per physical device or bus address.

## Error handling

Bus-facing methods return `IoResult`. Buffer contents remain under caller control after an error.

Callers shall inspect every returned `eurorack::io::IoResult` or `AnalogSample::result`. The driver does not log, throw exceptions, or silently perform unbounded retries.

## ISR, reentrancy, and concurrency

This driver is not reentrant. Blocking foreground API; not reentrant and unsuitable for ISR use. Serialize access when drawing and transferring may occur in different execution contexts.

Unless a future API explicitly states otherwise, this library makes no thread-safety guarantee. Shared buses must be arbitrated above the driver layer.

## Dependencies and resources

The library depends only on the granular framework interfaces declared by its `library.json`. Hardware resources are represented by injected adapters rather than selected globally. Refer to the [hardware page](../hardware/display/ssd1306.md) and the framework resource registry for concrete bus and GPIO requirements.

## Usage example

The executable example is located at `examples/components/ssd1306`:

```cpp
#include "../driver_example_support.hpp"
#include <array>
#include <eurorack/drivers/display/ssd1306.hpp>
int main(){ example::I2cBus bus; eurorack::drivers::display::Ssd1306 display(bus); std::array<std::uint8_t,1024U> pixels{}; eurorack::display::MonochromeCanvas canvas(pixels.data(),pixels.size(),128,64); canvas.applyPixel({0,0},eurorack::display::PixelOperation::Set); if(display.initialize()!=eurorack::io::IoResult::Success) return 1; return display.flush(canvas)==eurorack::io::IoResult::Success ? 0:1; }
```

The repository example, not this excerpt, is the build-checked source of truth.

## Limitations

- No internal scheduler, locking, or background transfer engine is provided.
- Electrical limits, decoupling, level compatibility, and analog accuracy remain hardware-design responsibilities.
- Unsupported silicon features are listed on the hardware page.
- API compatibility is not yet guaranteed because the framework remains alpha.
