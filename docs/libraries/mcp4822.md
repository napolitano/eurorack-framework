# MCP4822 driver

## Purpose

`eurorack-driver-mcp4822` is the software driver for **mcp4822**. This page documents the framework-facing contract. Device electrical behavior and board-level integration are documented separately in [docs/hardware/dac/mcp4822.md](../hardware/dac/mcp4822.md).

## Public API

- Public header: `eurorack/drivers/dac/mcp4822.hpp`
- Primary type: `Mcp4822`
- Maturity: `alpha`

- Two 12-bit channels with internal reference.
- Per-channel gain and shutdown control.
- Buffered state through `setCode()`, `setGain()`, and `setEnabled()`.
- `flushChannel()`, `flushBoth()`, and `latchOutputs()` support synchronized output changes.

The generated Doxygen reference remains authoritative for exact signatures, parameter types, enumerators, and return annotations.

## Lifecycle and configuration

Construct with SPI, CS, optional LDAC, and configuration. Buffer code, gain, and enable state per channel, then flush one or both channels and optionally latch outputs.

Constructors do not transfer ownership of framework adapters. Configuration values are captured by value unless the public header explicitly accepts a pointer or reference. No implicit global initialization is performed.

## Ownership and memory

SPI and GPIO dependencies are borrowed. The driver owns fixed-size arrays for two channels and performs no dynamic allocation.

Copy and move behavior follows the declarations in the public header. Applications should normally create one long-lived driver instance per physical device or bus address.

## Error handling

Flush methods return the first `IoResult` failure. Codes are stored as 12-bit values by the frame-building logic; callers should still pass intentional in-range values.

Callers shall inspect every returned `eurorack::io::IoResult` or `AnalogSample::result`. The driver does not log, throw exceptions, or silently perform unbounded retries.

## ISR, reentrancy, and concurrency

This driver is not reentrant. Not reentrant and not internally synchronized. Coordinate shared SPI access externally; do not call latch and transfer methods concurrently.

Unless a future API explicitly states otherwise, this library makes no thread-safety guarantee. Shared buses must be arbitrated above the driver layer.

## Dependencies and resources

The library depends only on the granular framework interfaces declared by its `library.json`. Hardware resources are represented by injected adapters rather than selected globally. Refer to the [hardware page](../hardware/dac/mcp4822.md) and the framework resource registry for concrete bus and GPIO requirements.

## Usage example

The executable example is located at `examples/components/mcp4822`:

```cpp
#include "../driver_example_support.hpp"
#include <eurorack/drivers/dac/mcp4822.hpp>
int main(){ example::SpiBus spi; example::DigitalOutput cs,ldac; eurorack::drivers::dac::Mcp4822 dac(spi,cs,&ldac); dac.setCode(eurorack::drivers::dac::Mcp4822Channel::A,1024U); dac.setGain(eurorack::drivers::dac::Mcp4822Channel::A,eurorack::drivers::dac::Mcp4822Gain::TwoX); return dac.flushBoth()==eurorack::io::IoResult::Success && spi.transfers()==2U ? 0:1; }
```

The repository example, not this excerpt, is the build-checked source of truth.

## Limitations

- No internal scheduler, locking, or background transfer engine is provided.
- Electrical limits, decoupling, level compatibility, and analog accuracy remain hardware-design responsibilities.
- Unsupported silicon features are listed on the hardware page.
- API compatibility is not yet guaranteed because the framework remains alpha.
