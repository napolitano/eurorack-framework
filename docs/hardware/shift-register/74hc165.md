# 74HC165

## Overview

Eight-bit parallel-in, serial-out shift register for sampled digital inputs.

| Property | Framework view |
|---|---|
| Manufacturer | Texas Instruments / compatible vendors |
| Framework library | `eurorack-driver-74hc165` |
| Public header | `<eurorack/drivers/shift/shift_register_74hc165.hpp>` |
| Primary class | `ShiftRegister74Hc165` |
| Hardware resources | SPI-compatible clock/data; parallel-load GPIO; optional cascade |
| Maturity | Unreleased Alpha |

## Key electrical and protocol characteristics

| Characteristic | Practical meaning |
|---|---|
| Function | Parallel-in, serial-out input expansion |
| Width | Eight inputs per device; devices may be cascaded |
| Capture | Active-low parallel-load pulse snapshots all inputs |
| Serial transfer | Clocked read compatible with the framework SPI abstraction |
| Output storage | The driver caches the most recent sampled chain |

## Framework operation sequence

Call `sample()` to capture and shift the complete chain. Query logical inputs from the cached result with `input()`. A failed transfer leaves the previous cached sample potentially stale.

## Framework support

Supported: One or more cascaded devices, explicit sampling and indexed access to the captured input state.

Not currently supported: Device-specific diagnostics, readback or uncommon special modes are not exposed unless listed above.

## Dependencies and ownership

The library owns no global scheduler and performs no dynamic dispatch beyond the framework I/O interfaces. The consuming application owns the bus and GPIO adapter objects and must ensure that their lifetime exceeds the driver lifetime. Bus arbitration follows the `IoResult` contract; calls may report Busy or transport errors.

Hardware resources: **SPI-compatible clock/data; parallel-load GPIO; optional cascade**. Sharing a bus is supported only when each device has independent selection/latch signals and all transfers occur through the same bus abstraction. Driver calls are foreground operations unless the application and platform adapter explicitly document ISR safety.

## Minimal code example

```cpp
#include <eurorack/drivers/shift/shift_register_74hc165.hpp>

ShiftRegister74Hc165 inputs(spi, parallelLoad, 1U);
if (inputs.sample() == eurorack::io::IoResult::Success) {
    const bool first = inputs.input(0U);
}
```

The variables representing buses, pins, delays and canvases are platform adapters supplied by the application. Check every returned `IoResult` in production code.

## Integration notes

1. Verify supply voltage, logic thresholds, reference voltage and output compliance against the exact part and package.
2. Initialize selection, latch, blanking or enable pins to a safe inactive state before enabling the peripheral.
3. Keep bus transactions outside time-critical ISRs unless a platform-specific contract explicitly permits them.
4. Add decoupling and analog filtering according to the manufacturer recommendations and the required noise performance.
5. Validate the implementation on target hardware; native tests prove protocol formation and state handling, not analog performance.
