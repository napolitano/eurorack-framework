# TLC5916

## Overview

Eight-channel constant-current LED sink driver with serial data interface.

| Property | Framework view |
|---|---|
| Manufacturer | Texas Instruments |
| Framework library | `eurorack-driver-tlc5916` |
| Public header | `<eurorack/drivers/led/tlc5916.hpp>` |
| Primary class | `Tlc5916` |
| Hardware resources | SPI-compatible serial bus; latch; output enable; external current-setting resistor |
| Maturity | Unreleased Alpha |

## Why this device is supported

The device covers a recurring embedded/Eurorack hardware task while remaining independently selectable. Projects that do not use it do not need to add its library or transitive dependencies.

## Key electrical and protocol characteristics

| Characteristic | Practical meaning |
|---|---|
| Outputs | Eight constant-current sink channels per device |
| Dimming model | The current driver exposes binary off/on state, not per-channel PWM |
| Current programming | One external resistor programs sink current |
| Cascading | Multiple devices form one serial chain |
| Output enable | Optional active-low OE allows global blanking |

## Pin and signal integration

| Device signal | Framework dependency | Integration requirement |
|---|---|---|
| CLK / SDI | `SpiBus` | Shift one bit per logical channel |
| LE | `DigitalOutput` | Latches shifted data into the output register |
| /OE | Optional `DigitalOutput` | Use to guarantee dark startup and global blanking |
| R-EXT | External resistor | Select current from the manufacturer equation and thermal limits |

## Framework operation sequence

Set logical channel brightness to zero or nonzero, then call `flush()`. Any nonzero 16-bit framework brightness becomes an enabled constant-current channel because this device does not provide per-channel grayscale PWM.

## Framework support

Supported: Cascaded devices, per-channel on/off mapping through the LedBank brightness contract, clear, flush and output enable.

Not currently supported: Diagnostic special modes and programmable global-current commands are not exposed.

## Dependencies and ownership

The library owns no global scheduler and performs no dynamic dispatch beyond the framework I/O interfaces. The consuming application owns the bus and GPIO adapter objects and must ensure that their lifetime exceeds the driver lifetime. Bus arbitration follows the `IoResult` contract; calls may report Busy or transport errors.

Hardware resources: **SPI-compatible serial bus; latch; output enable; external current-setting resistor**. Sharing a bus is supported only when each device has independent selection/latch signals and all transfers occur through the same bus abstraction. Driver calls are foreground operations unless the application and platform adapter explicitly document ISR safety.

## Minimal code example

```cpp
#include <eurorack/drivers/led/tlc5916.hpp>

Tlc5916 leds(spi, latch, 1U, &outputEnable);
leds.setBrightness(0U, 0xFFFFU);
const auto result = leds.flush();
```

The variables representing buses, pins, delays and canvases are platform adapters supplied by the application. Check every returned `IoResult` in production code.

## Integration notes

1. Verify supply voltage, logic thresholds, reference voltage and output compliance against the exact part and package.
2. Initialize selection, latch, blanking or enable pins to a safe inactive state before enabling the peripheral.
3. Keep bus transactions outside time-critical ISRs unless a platform-specific contract explicitly permits them.
4. Add decoupling and analog filtering according to the manufacturer recommendations and the required noise performance.
5. Validate the implementation on target hardware; native tests prove protocol formation and state handling, not analog performance.
