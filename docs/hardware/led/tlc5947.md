# TLC5947

## Overview

24-channel constant-current LED sink driver with autonomous 12-bit grayscale PWM.

| Property | Framework view |
|---|---|
| Manufacturer | Texas Instruments |
| Framework library | `eurorack-driver-tlc5947` |
| Public header | `<eurorack/drivers/led/tlc5947.hpp>` |
| Primary class | `Tlc5947` |
| Hardware resources | SPI; XLAT; optional BLANK; external current-setting resistor |
| Maturity | Unreleased Alpha |

## Why this device is supported

The device covers a recurring embedded/Eurorack hardware task while remaining independently selectable. Projects that do not use it do not need to add its library or transitive dependencies.

## Key electrical and protocol characteristics

| Characteristic | Practical meaning |
|---|---|
| Outputs | Twenty-four constant-current sink channels per device |
| PWM resolution | 12-bit grayscale PWM per channel |
| Interface | Clocked serial data with a separate latch; no returned data |
| Cascading | Frame size grows by 36 bytes per device |
| Output enable | Active-low BLANK/OE may guarantee dark startup |

## Pin and signal integration

| Device signal | Framework dependency | Integration requirement |
|---|---|---|
| CLK / DIN | `SpiBus` | Transfer the complete packed grayscale frame |
| LAT | `DigitalOutput` | Pulse after transfer to update all PWM registers |
| /OE | Optional `DigitalOutput` | Recommended for controlled startup and blanking |
| R-EXT | External resistor | Defines sink current; verify total package dissipation |

## Framework operation sequence

Provide caller-owned value and frame buffers sized for the complete chain, verify `valid()`, call `initialize()`, update logical brightness values, and call `flush()`. Visible startup modes additionally require a `DelayProvider`.

## Framework support

Supported: Cascaded devices, 12-bit PWM packing, caller-owned buffers, safe initialization, silent/sequential/all-flash startup policies and BLANK control.

Not currently supported: Device-specific diagnostics, readback or uncommon special modes are not exposed unless listed above.

## Dependencies and ownership

The library owns no global scheduler and performs no dynamic dispatch beyond the framework I/O interfaces. The consuming application owns the bus and GPIO adapter objects and must ensure that their lifetime exceeds the driver lifetime. Bus arbitration follows the `IoResult` contract; calls may report Busy or transport errors.

Hardware resources: **SPI; XLAT; optional BLANK; external current-setting resistor**. Sharing a bus is supported only when each device has independent selection/latch signals and all transfers occur through the same bus abstraction. Driver calls are foreground operations unless the application and platform adapter explicitly document ISR safety.

## Minimal code example

```cpp
#include <eurorack/drivers/led/tlc5947.hpp>

std::array<std::uint16_t, 24U> values{};
std::array<std::uint8_t, 36U> frame{};
Tlc5947 leds(spi, latch, values.data(), values.size(),
           frame.data(), frame.size(), 1U, &blank);
if (leds.initialize() == eurorack::io::IoResult::Success) {
    leds.setBrightness(0U, 0x8000U);
    leds.flush();
}
```

The variables representing buses, pins, delays and canvases are platform adapters supplied by the application. Check every returned `IoResult` in production code.

## Integration notes

1. Verify supply voltage, logic thresholds, reference voltage and output compliance against the exact part and package.
2. Initialize selection, latch, blanking or enable pins to a safe inactive state before enabling the peripheral.
3. Keep bus transactions outside time-critical ISRs unless a platform-specific contract explicitly permits them.
4. Add decoupling and analog filtering according to the manufacturer recommendations and the required noise performance.
5. Validate the implementation on target hardware; native tests prove protocol formation and state handling, not analog performance.
